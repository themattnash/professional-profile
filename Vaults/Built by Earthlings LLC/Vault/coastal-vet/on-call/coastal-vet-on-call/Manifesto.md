I. Clear Requirements & Vision
Every project begins with a formally defined, versioned specification that explicitly states the problem being solved, the users and system actors involved, and the measurable definition of success; The problem must include context, constraints, impact, and explicit non-goals; All users and actors must be enumerated and mapped to functional requirements — including AI agents, whose capabilities, trust levels, and interaction boundaries must be defined with the same rigor as human actors; Success criteria must be quantifiable, testable, and observable (functional and non-functional); Ambiguity is unacceptable — if it is not written in the specification, it does not exist, and if it cannot be measured, it is not complete; Specifications used to drive AI-assisted development must follow machine-readable conventions: discrete labeled requirements with unambiguous pass/fail acceptance criteria, bounded scope, and zero tolerance for unresolved placeholders at execution time — ambiguity that a human engineer resolves from context becomes a source of hallucination for an AI agent and must be resolved before planning begins.

II. User-Centered Design
Software must be designed to deliver measurable value to clearly defined users; All features must solve validated user problems rather than assumed needs; Usability is a requirement, not an afterthought — workflows and interfaces must minimize cognitive load, friction, and error rates; Accessibility is mandatory — systems must support assistive technologies, inclusive design standards, and diverse user capabilities; Designs must account for failure states (network errors, partial data, timeouts, service degradation) and provide clear recovery paths; Solutions must function under constrained environments including low-bandwidth or unstable connections; User feedback must be gathered early and continuously and incorporated into the specification; No feature may be implemented without a defined user benefit and traceable problem alignment; If a solution does not improve user outcomes, resilience, or accessibility, it must be reconsidered or removed.

III. Simplicity (KISS)
Systems must be designed and implemented using the simplest solution that fully satisfies the specification; Unnecessary abstractions, premature optimizations, speculative extensibility, and clever but opaque implementations are prohibited; Clarity is prioritized over ingenuity — code, architecture, and workflows must be understandable by a competent engineer without tribal knowledge; Every added dependency, layer, or pattern must have explicit justification tied to a requirement; Complexity must be intentional, measurable, and minimized, as unmanaged complexity increases defect rates, operational risk, onboarding cost, and long-term maintenance burden; In AI-assisted workflows, complexity carries an additional cost: a pattern a competent engineer recognizes by convention must be explicitly stated for an agent operating only from provided context — any architectural choice not expressible as a clear rule in the project constitution or agent context file is presumptively unjustified and must be challenged.

IV. Modularity & Separation of Concerns  
Systems must be decomposed into small, cohesive, independently testable components with clearly defined responsibilities and explicit interfaces; Each module shall have a single, well-defined purpose and must isolate business logic, presentation, data access, and infrastructure concerns; Internal implementation details must not leak across module boundaries, and inter-module communication must occur through stable, documented contracts; Modules must be independently testable and deployable where feasible; Changes in one module must not cause cascading modifications in unrelated components; This structural discipline enables easier testing, simplified maintenance, controlled scalability, parallel team development, and safer long-term evolution of the system.

V. Maintainability & Clean Code
Code must be written for long-term clarity and sustainability, prioritizing readability over brevity and maintainability over short-term speed; Structure shall reflect clear intent with logical organization, consistent naming conventions, and predictable patterns; Every module, class, and function must have a single clear responsibility and minimal side effects; Documentation must explain purpose and intent (the “why”), not restate obvious implementation details; Code formatting, architectural conventions, and style guidelines must be consistently enforced across the project; Technical debt must be tracked explicitly rather than hidden; The system must remain understandable by engineers unfamiliar with its original authorship, ensuring safe onboarding, refactoring, debugging, and long-term evolution; In AI-assisted workflows, code conventions must be expressed in machine-consumable agent context files maintained alongside source code — conventions that exist only in tribal knowledge or informal documentation are invisible to agents and will not be applied; agent context files are living artifacts subject to the same version discipline as the codebase they describe.

VI. Testing & Quality Assurance
Quality is mandatory and must be enforced through systematic, automated, and continuous validation practices; Every requirement must be verifiable through defined test coverage including unit tests (isolated logic validation), integration tests (cross-module interaction validation), and system-level tests where applicable; Automated testing is required and must be integrated into the build and deployment pipeline; Continuous testing must occur on every commit, merge, and release candidate to prevent regression; Test coverage must map directly to specification requirements and acceptance criteria; No feature is considered complete without corresponding tests; Testing enables defect reduction, safe refactoring, architectural stability, and predictable, reliable releases; Manual testing may supplement but must not replace automated verification; Untested code is considered incomplete and introduces unacceptable systemic risk; AI-generated code must be treated as untrusted output until verified against acceptance criteria defined before generation — tests written after the fact against AI output do not constitute independent verification; pre-execution quality gates in agentic workflows are mandatory stops, not advisory checkpoints, and no phase may begin until prior gates are explicitly cleared.

VII. Iterative & Incremental Development
Work must be delivered in small, validated increments rather than as a single large release; Each increment shall produce a working, integrated slice of functionality aligned to the specification and acceptance criteria; Releases should occur early and regularly to expose integration issues, usability gaps, and requirement misunderstandings while they are still cheap to fix; Feedback from real usage and review must continuously refine subsequent iterations; Plans and designs may evolve, but scope changes must be reflected as explicit spec revisions with traceable rationale; In agentic workflows, each task or user story must be structured as an independently executable, verifiable unit with explicit completion criteria — an agent that cannot determine when a task is done will over-generate or halt incorrectly; bounded scope with observable pass/fail criteria prevents both failure modes.

VIII. Risk Management  
Risks must be identified, documented, and reduced as early as possible, with explicit ownership and mitigation plans; Technical, business, performance, operational, and security risks must be tracked in the spec or project artifacts and revisited continuously; High-impact or high-uncertainty items must be de-risked first via spikes, prototypes, proofs-of-concept, or early implementation of critical paths; Unknowns must be converted into knowns through measurable experiments; Work sequencing should prioritize eliminating failure modes before optimizing or expanding features.

IX. Version Control & Collaboration
All work must be tracked in version control (e.g., Git) with disciplined collaboration practices; Every change must be reviewable, attributable, and reversible via branching, pull requests, and a documented merge strategy; Commit history must preserve intent and support debugging, auditing, and release reconstruction; Collaboration workflows must enable parallel development without destabilizing mainline code; Rollback and hotfix procedures must be supported as first-class operational capabilities; When AI agents author or generate changes, each commit must be attributable to both the human who authorized the task and the agent that executed it, with commit messages referencing the task ID and specification artifact that drove the change — this preserves the traceability chain from requirement through spec to code and enables auditing of agent behavior across the project's history.

X. Documentation & Communication  
Communication must be deliberate, continuous, and backed by durable documentation shared across engineering, product, design, and QA; Documentation must capture architecture, interfaces, setup/runbooks, and key decisions (including tradeoffs and rationale), not just outcomes; Specs, diagrams, and API contracts must be treated as living artifacts and updated alongside code changes; Decisions made in conversation are not real until recorded in an agreed artifact; Misalignment must be surfaced early through reviews, demos, and clear acceptance criteria.

XI. Scalability & Performance Awareness  
Systems must be designed with expected growth and operational constraints in mind, with performance treated as a measurable requirement; Capacity targets (e.g., concurrency, throughput, data volume) and latency budgets must be defined and tested against realistic workloads; Design must enable scaling paths (horizontal and/or vertical) appropriate to the domain, and avoid bottlenecks that block growth; Performance optimizations must be driven by measurements and profiling rather than assumptions; The system must degrade gracefully under load and remain observable for tuning.

XII. Security by Design  
Security must be built into requirements, architecture, and implementation from the beginning, not added after features are complete; Authentication and authorization must be explicit and least-privilege by default; Input validation, secure defaults, secrets handling, and encryption (in transit and at rest where appropriate) are mandatory; Threat modeling must be performed for critical flows and updated as the system evolves; Security testing and reviews must be integrated into development workflows; Vulnerability remediation must be tracked, prioritized, and verifiable.

XIII. Continuous Integration & Deployment (CI/CD)
Build, test, and deployment processes must be automated, repeatable, and reliable; Every change must pass automated checks (linting, tests, security checks where applicable) in CI before merge; Deployments must be standardized and scripted to reduce human error, support frequent releases, and enable rapid rollback; Release pipelines must provide clear signals (pass/fail, artifact versioning, provenance) and preserve traceability from spec to code to deployed artifact; Automation should shorten feedback loops, increase confidence, and improve delivery cadence without sacrificing quality; In spec-driven AI pipelines, CI encompasses the full artifact chain — a specification change must trigger re-validation of dependent plans, task lists, and agent context files before implementation proceeds; traceability from spec to deployed artifact is not merely a desirable property but the primary audit mechanism for verifying that agents implemented what was specified and nothing more.

XIV. AI Agent Governance
AI agents participating in development workflows operate under explicitly defined, versioned constitutions that establish non-negotiable constraints before any implementation begins; no agent may take an action not sanctioned by the current constitution; agent authority is explicitly bounded by scope — agents must operate in dry-run or audit mode by default for irreversible actions (deploys, deletions, external API calls, financial transactions) and must surface the planned action for human approval before execution; the project constitution is a ratified artifact, not a prompt, and amendments require documented rationale, impact assessment across dependent templates, and an explicit version increment; an agent that cannot locate or parse a valid constitution must halt and request clarification rather than proceeding on assumed defaults; human approval is non-negotiable at: specification ratification, architectural decisions, security-sensitive changes, production deployments, and any action that cannot be reversed; AI-generated artifacts — code, tests, documentation, decisions — are untrusted output until reviewed and accepted by a human with authority over the relevant scope.

XV. Prompt and Context Integrity
The instructions, context, and constraints provided to an AI agent constitute a contract: the agent implements exactly what the context specifies, including its omissions; agent context files must be treated as first-class project artifacts — versioned, reviewed, and updated as part of every planning phase — not as one-time scaffolding; a stale or incomplete context file is a defect with the same severity as a stale interface contract — it will cause the agent to produce outputs inconsistent with current project requirements without any visible error signal; context drift, where an agent's operating context diverges from ground truth, is a class of defect that must be detected and corrected; the provenance of every agent instruction must be traceable to a spec, constitution, or explicitly approved addition — context that cannot be attributed to a source must be removed; critical state must persist in durable, human-readable artifacts rather than ephemeral agent memory; prompts and agent instructions are documentation artifacts subject to the same versioning and review requirements as code.
